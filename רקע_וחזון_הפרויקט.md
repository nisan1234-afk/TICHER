# כיתה פלוס (Kita Plus) — רקע, חזון ומטרות

מסמך זה נועד לתת לקלוד קוד (או לכל מפתח/סוכן שממשיך את הפרויקט) את כל ההקשר שנצבר בשיחות עם ניסן, מעבר לקוד עצמו. המסמך הטכני-תפעולי (מבנה, מוסכמות, מלכודות) נמצא ב-CLAUDE.md; כאן — ה"למה".

## מי זה ניסן ומה ההקשר

ניסן הוא מורה בתיכון ישיבתי בישראל, ומריץ מגמת תיירות דיגיטלית לכיתות י'. הפרויקט הזה הוא התשתית הטכנולוגית שהוא בונה כדי לתמוך בהוראה שלו.

**חלוקת אחריות:** ניסן מקבל את כל ההחלטות הפדגוגיות והניהוליות של הפרויקט. ההחלטות הטכניות נתונות לשיקול דעתו העצמאי של קלוד. ניסן מעדיף לעבוד לאט ובזהירות, ודוחף בחזרה כשקלוד מתקדם מהר מדי או מניח הנחות בלי לבדוק.

## מה זה כיתה פלוס — החזון

**כיתה פלוס** היא פלטפורמת ניהול לימודי מקיפה, לא רק כלי חד-פעמי. הרעיון הארכיטקטוני המרכזי: הפלטפורמה בנויה כמודולרית לגמרי — כל מקצוע/שירות הוא ריפו ו-Google Sheet עצמאיים, מחוברים דרך API מרכזי אחד ומערכת הרשאות משותפת. כך אפשר להוסיף בעתיד מקצועות/מגמות נוספות בלי לשכתב את הליבה.

**המקצוע הראשון שנבנה על הפלטפורמה** הוא **תיירות דיגיטלית ("tourismdigi")** — קורס מבוסס-פרויקט לכיתה י'.

## השיטה הפדגוגית של קורס התיירות הדיגיטלית

קבוצות תלמידים בוחרות אתר תיירות אמיתי (קיים באינטרנט) ומנתחות את הנוכחות הדיגיטלית שלו לאורך השנה, דרך **8 מדורים** (sections):

1. פרטי האתר
2. תיאור כללי
3. נוכחות דיגיטלית
4. מושגי תיירות דיגיטלית
5. חוויית משתמש
6. חוזקות דיגיטליות
7. הצעות לשיפור — **המדור המשוקלל ביותר** בהערכה
8. סיכום אישי

בסוף השנה מתקיימת **תערוכה פומבית** של כל אתרי הקבוצות — זהו מנגנון המוטיבציה המרכזי של הקורס (התלמידים יודעים שהעבודה שלהם תוצג לקהל).

**מיפוי לתוכנית הלימודים:** תשע יחידות לימוד של משרד החינוך ממופות לשמונת המדורים. חלק מהיחידות מזינות ישירות את תוצר התלמיד (האתר שהם בונים/מנתחים), וחלק הן יחידות תיאורטיות עצמאיות שנבדקות בנפרד אחרי פורים. התוצר המרכזי אמור להסתיים לפני פסח; היחידות התיאורטיות ממשיכות עד סוף השנה.

## החלטות ארכיטקטוניות שכבר סוכמו (לא לשנות בלי לדבר עם ניסן)

- **התחברות:** Google Sign-In בלבד, ללא סיסמאות. האימייל הוא המזהה האוניברסלי של המשתמש.
- **Backend:** Google Apps Script יחיד ומרכזי (לא מפוצל לכמה סקריפטים).
- **מודולריות:** כל מקצוע/שירות הוא ריפו ו-Sheet נפרדים.
- **תפקידים (roles):** חמישה — admin, school_admin, teacher, homeroom, student. משתמש יכול להחזיק כמה תפקידים במקביל; ההקצאה נעשית רק ע"י admin, בלשונית `roles` ייעודית.
- **Homeroom** (מחנך) הוא תפקיד צפייה בלבד — אינו יכול ליצור קשר ישיר עם תלמידים.
- **דשבורד מורה** מציג דינמית רק את הכפתורים/הטאבים הרלוונטיים לתפקידים של המשתמש המחובר.
- **מודל קבוצות (v2):** עברנו מ"זוגות" נוקשים (student1/student2) למודל גמיש של קבוצות בכל גודל, עם שדה `members` (מופרד בפסיקים) ושם קבוצה חופשי שהמורה בוחר.

## פילוסופיית הצ'אטבוט

עוזר בפאנל צד שמתערב **רק כשתלמידים תקועים** — לא אמור להיות המנוע המרכזי של הלמידה. אם הוא נכשל באמצע שיעור, ההוראה ממשיכה בדרך המסורתית בלי תלות בו. זו נקודה עקרונית: הטכנולוגיה תומכת, לא מחליפה, את ההוראה.

## לקחים ועקרונות מרכזיים (חשוב לזכור!)

- **פריסת Apps Script:** תמיד להשתמש ב"New Deployment" (לא "manage existing") אחרי שינויי קוד. שימוש חוזר ב-endpoint קיים מגיש את הגרסה הישנה מהקאש בלי שום שגיאה. אם מתקבל HTML גולמי במקום JSON — כמעט תמיד סימן שהסקריפט נמחק, לא נשמר, או שהפריסה לא עדכנית. **זו לא בעיית CORS**, למרות שזה נראה ככה.
- **עריכת GitHub Pages:** עריכה ישירה בעורך הווב של GitHub אמינה משמעותית יותר מהורדה-ואז-העלאה מחדש (כמה תקריות rollback קרו עם השיטה השנייה). לוקח 1-3 דקות לקומיטים להשתקף; צריך hard refresh או גלישה בסתר כדי לעקוף קאש.
- **Google Drive MCP:** יכול לקרוא תוכן ומטא-דאטה של קבצים אבל לא לכתוב — כל פעולת כתיבה חייבת לעבור דרך Apps Script. רשימת תוכן תיקייה לפי folder ID הייתה לא אמינה; הפתרון היה להשתמש ב-`search_files` עם query מהצורה `'FOLDER_ID' in parents`.
- **Google Sign-In:** דורש `use_fedcm_for_prompt: true` ב-`google.accounts.id.initialize` כדי להימנע מאזהרות deprecation של FedCM.
- קישורי Canva ודפים עם רינדור JavaScript לא ניתנים ל-scraping — דורשים העתקה-הדבקה ידנית או ייצוא קובץ.
